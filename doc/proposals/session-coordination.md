## Problem
In multi-agent collaborative sessions, multiple agents (e.g. Claude, Codex, Gemini/agy) and the human user (Roberto) may work concurrently in different git worktrees or out-of-band environments. If they write to a single shared `session.md` file, they encounter:
1. **File Write Collisions:** The local files are modified concurrently, leading to write conflicts (e.g., "file has been modified since read").
2. **Lost Turns / Merging Overhead:** Git merge conflicts on `session.md` force manual rebasing or force Roberto to back-fill steering turns out of order.
3. **Control Plane vs. Data Plane Confusions:** While the Collab MCP handles sequential turn-taking at the communication layer, it does not prevent collisions if agents work in parallel or edit offline.

---

## Proposed UX / Behavior
We propose a **Log-Structured Merge (LSM) Session Coordination** pattern:

### 1. Split Session Files (Memtables)
- Each participant edits only their own dedicated file under the session directory: `session.$BOTNAME.md` (e.g., `session.agy.md`, `session.claude.md`, `session.codex.md`).
- Roberto writes directly to `session.roberto.md` when contributing out-of-band. Relayed human steering is logged inside each agent's file.
- Because each file has exactly one writer, there is zero write contention on these files.

### 2. Standardized Time-Stamped Entries
- Every entry in `session.*.md` must start with a strict RFC3339 header:
  ```markdown
  ## [handle, YYYY-MM-DDTHH:MM:SS+-HH:MM]
  ```
- Timestamps must include timezone offsets to ensure correct chronological sequencing.
- Session closeout should be recorded inside a participant's source file rather than as a synthetic `## [Closed, ...]` participant entry. If a generated close marker is needed, the merge script reserves `Closed` as a non-participant handle.

### 3. Compaction Script (`tools/merge_sessions.py`)
- We implement a compilation utility `tools/merge_sessions.py` that reads all `session.*.md` files in a session directory.
- The script parses headings, normalizes all timestamps to UTC, and merges them using a stable k-way merge sort.
- The tie-breaker for identical timestamps is `(utc_instant, source_file, original_order)`.
- The unified `session.md` is compiled with a banner warning (`<!-- GENERATED by tools/merge_sessions.py - DO NOT EDIT. Edit session.<bot>.md instead. -->`) and a list of active participants, followed by the sorted entries.

### 4. Live Coordination and Final Compaction
- During the live consensus loop, agents do **not** repeatedly regenerate `session.md`. It is allowed to be stale while the session is active.
- Agents watch each other's split logs directly through `git diff`, `git status`, `tail`, editor refresh, or collab MCP messages. The split files are the live coordination surface.
- Once consensus is reached, exactly one closer runs `tools/merge_sessions.py <session-dir>` to compact the split logs into `session.md`.
- A pre-commit hook may run `tools/merge_sessions.py <session-dir> --check` to catch stale generated logs before commit. If it fails, one agent or Roberto runs the merge once, then retries the check.

### 5. Composition with Collab MCP
- This data-plane log layout does *not* replace the Collab MCP. The Collab MCP acts as the control plane to queue and assign parallel tasks or handle turn handoffs. The LSM-style logs handle the data recording to ensure no commits ever block or collide.

## Validation Plan
1. **Live Coordination Verification:** During a session, have each agent append only to its own `session.$BOTNAME.md`, then watch peer updates through `git diff`, `git status`, `tail`, editor refresh, or collab MCP messages without regenerating `session.md`.
2. **Final Compaction Verification:** After consensus, have one closer run `tools/.venv/bin/python tools/merge_sessions.py doc/collab/sessions/2026-06-08-session-management` once to verify it correctly parses and merges the split logs.
3. **Freshness Verification:** Run the script with `--check` to verify correct exit codes when `session.md` matches or does not match.
4. **Cross-Timezone & Sorting Verification:** Confirm that timestamps with offsets (e.g., `+02:00`) normalize and sort correctly against UTC.

---

## Product-Fit Notes
- Using split logs significantly reduces git-merge friction in multi-agent workflows.
- Exposing `--check` prevents committing out-of-sync logs.
- The workflow scales gracefully to any number of participants.
