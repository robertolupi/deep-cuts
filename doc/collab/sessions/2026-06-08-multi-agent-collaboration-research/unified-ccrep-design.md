# Unified CCREP 2.1: Cognitive Consensus and Result Enrichment Protocol (Real-World Sandbox Design)

## 1. Executive Summary & Design Constraints
Unified CCREP 2.1 is designed around the physical constraints of local sandboxed environments (macOS/Unix), Git repository semantics, and Model Context Protocol (MCP) clients. 

A critical vulnerability in previous agent coordination models is the assumption that the system can dynamically adjust the LLM's inference temperature ($T$) or parsing parameters mid-session. In practice:
* **Temperature is Client-Configured**: Agent execution hosts or clients (e.g., Claude Desktop, VS Code plugins) configure temperature at startup. The running agent has no direct API or capability to dynamically alter its own temperature during a task iteration.
* **Sampling APIs are Optional**: The MCP `sampling/createMessage` primitive allows requesting completions with specific temperatures, but many runtime clients clamp, override, or fail to implement this endpoint.

Unified CCREP 2.1 replaces these unhonored prompt constraints with **Hard Physical and Syntactic Constraints** enforced programmatically by the MCP server's file operations, AST checkers, git worktrees, and local database lock-gates.

---

## 2. Hard Physical Constraints & Syntactic Gates

Instead of asking the agent to "be less creative" at late revision rounds, the MCP server physically limits the changes it can commit.

```
       [Revision Progress]
       Round 0 ──────────────> Round 1 ──────────────> Round 3 ──────────────> Round 5 (Hard Cap)
       [Scope: Global]         [Scope: Local AST]      [Scope: Surgical Diff]  [Scope: Lockout]
       Any files               No signature changes    Max 2 files, 15 lines   Only linter-directed
       Unlimited diff          Internal body changes   Immediate test scope    compilation fixes
```

### A. AST & Line-Diff Gatekeeping
The MCP write/commit tools check the proposed git diff at each revision round $n$:
1. **Round 0 (Exploratory)**: No constraints. The agent can modify any file, swap dependencies, or refactor core architectures.
2. **Round 1 & 2 (Refinement)**: The system parses the modified files' ASTs (using `tree-sitter` or local language parsers).
   * *Rule*: The diff must not introduce new function definitions, class declarations, or change existing signature interfaces. It must only modify logic *inside* existing function/method bodies.
3. **Round 3 & 4 (Surgical Repair)**: The MCP server enforces a strict character and line budget:
   * *Rule*: The diff must touch at most $2$ files and contain at most $15$ added/modified lines of code.
   * *Rule*: The changes must be localized within $3$ AST blocks of the functions containing the failing assertions.
4. **Round 5 (Terminal)**: The gatekeeper rejects all manual code changes.
   * *Rule*: Only exact, compiler-suggested fixes (e.g. `rustc --fix`, `eslint --fix`) or diffs automatically generated by the Codex linter tool are allowed.

*If any agent proposal violates the gate for the current round, the MCP server rejects the write operation immediately with a descriptive error:*
`PROPOSAL REJECTED: Round 3 requires surgical edits (max 15 lines, 2 files). Your change modified 48 lines across 3 files.`

### B. Edit-War & Oscillation Lockout
* **Detection**: The server computes the normalized Levenshtein distance ($D_L$) on the code diff between revision $n$ and $n-2$. If $D_L < 0.05$ and the authoring agents alternate (Claude $\rightarrow$ Gemini $\rightarrow$ Claude), the server triggers an **Oscillation Lockout**.
* **Action**: The two alternating agents are locked out of the task. A third agent (e.g., Codex) is woken up with the prompt:
  `A style oscillation has been detected between Claude and Gemini. You are tasked with writing a compromise diff that resolves the compile error using the minimal change possible.`

## 3. Git Worktree Isolation & Test Orchestration

Running test suites or compilation checks on the active developer workspace causes state corruption, locks files, and breaks local developer workflows. CCREP 2.1 isolates all evaluations using Git Worktrees.

```
                  [Developer Workspace: main]
                              │
                    (submit_proposal tool)
                              │
                              ▼
            [Create Worktree under .mcp/worktrees/]
            git worktree add --detach .mcp/worktrees/prop-001 feat/branch
                              │
                              ▼
             [Execute Codex Verification Harness]
             (Builds, lints, and test suites run here)
                              │
                              ▼
           [Collect MOOV Quality Metrics & Discard]
           git worktree remove --force .mcp/worktrees/prop-001
```

### Worktree Lifecycle
1. **Creation**: When `submit_proposal()` is called, the MCP server runs:
   ```bash
   git worktree add --detach .ccrep/worktrees/{proposal_id} {commit_sha}
   ```
   This creates a lightweight, separate checkout directory referencing the repository object database without copying files on disk.
2. **Execution**: The verification harness executes commands (e.g., `cargo test` or `npm run test`) within the worktree directory.
3. **Cleanup**: Once quality metrics are written to the database, the server runs:
   ```bash
   git worktree remove --force .ccrep/worktrees/{proposal_id}
   ```
   This deletes the directory and cleans up the worktree reference, leaving the developer's main branch and working copy completely untouched.

## 4. Local Database & Concurrency Serialization

CCREP 2.1 relies on a local SQLite database (`.ccrep/state.db`) for transaction logging. Because WSL2/DrvFS mounts on Windows and BTRFS COW filesystems do not support standard SQLite Write-Ahead Logging (WAL) shared-memory files (`*.db-shm`), the server implements a physical locking fallback.

```
                       [MCP Server Transaction]
                                  │
                                  ▼
                     [Atomic lock: mkdir locks/db.lock]
                       /                      \
               (Success)                      (Collision/EEXIST)
                   │                                  │
                   ▼                                  ▼
         [Open SQLite Database]               [Randomized Backoff]
      (PRAGMA journal_mode = WAL)                 (Retry loop)
                   │
                   ▼
       [Commit Proposal/Critique]
                   │
                   ▼
         [Release Lock: rmdir]
```

### Locking Operations
* **Primary Lock**: Before opening the SQLite connection for write operations, the server attempts to create a lock directory:
  ```python
  Path(".ccrep/locks/db.lock").mkdir(parents=True, exist_ok=False)
  ```
* **Collision Handling**: If it raises `FileExistsError` (`EEXIST`), the lock is held. The process enters a randomized exponential backoff retry loop (10ms to 150ms) to serialize database writes.
* **Database Configuration**:
  ```sql
  PRAGMA journal_mode = WAL;
  PRAGMA synchronous = NORMAL;
  PRAGMA busy_timeout = 5000;
  ```
* **Release**: After the transaction commits, the server removes the lock directory:
  ```python
  Path(".ccrep/locks/db.lock").rmdir()
  ```

## 5. Daemonless Wake-Up Signaling (Doorbells)

To avoid running active background daemon processes, CCREP 2.1 utilizes local POSIX named pipes (FIFOs) or directory watchers to notify agents when it is their turn to act.

```
       [MCP Scribe/Orchestrator] ──> Writes message to spool/gemini/new/
                                 ──> Touches Named Pipe: echo "1" > .ccrep/doorbells/gemini
                                                               │
                                                               v
                                                [Gemini Agent wakes up from blocking read]
                                                cat .ccrep/doorbells/gemini
```

### Signaling Sequence
1. **Delivery**: The orchestrator writes a structured JSON envelope to `spool/<agent>/new/<uuid>.json`.
2. **Doorbell Ring**: The orchestrator writes a single byte to the agent's dedicated FIFO pipe:
   ```bash
   echo "1" > .ccrep/doorbells/{agent_name}
   ```
3. **Execution**: The target agent (which has been blocking on a read of its doorbell FIFO) is unblocked, reads the new envelope from its spool directory, executes its turn, and writes its output.

## 6. Multi-Objective Optimization Vector (MOOV)

Quality evaluation is based on a Multi-Objective Optimization Vector (MOOV):
$$\vec{Q} = [q_{\text{functional}}, q_{\text{coverage}}, q_{\text{complexity}}, q_{\text{performance}}]$$

All vector values are normalized $\in [0.0, 1.0]$. The consensus gate operates on **Pareto Dominance**:
* A proposal $A$ is considered *better* than proposal $B$ if it is not dominated by $B$ (i.e., $A$ is at least as good as $B$ in all objectives, and strictly better in at least one).
* If a new revision degrades *any* metric (e.g., increases cyclomatic complexity or decreases code coverage) even if it improves the test score, it is flagged for critique and cannot be merged without explicit override.

### Laplacian-Smoothed Weighted Quorum
The voting weight for agent $a_i$ in domain $D$ is calculated using Laplace-smoothed historical verdict accuracy $\beta_{i,D}$:
$$\beta_{i,D} = \frac{N_{\text{correct}} + 1}{N_{\text{total}} + 2}$$
$$w_{i,D} = \ln\left( \frac{\beta_{i,D}}{1 - \beta_{i,D}} \right)$$

* **Quorum Criteria**: Sum of Approver weights $\ge 1.5$, including at least one agent with $w_{i,D} \ge 0.5$ in the matching domain.

    """Acquires a physical directory lock to serialize database writes."""
