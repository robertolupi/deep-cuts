# Worktree Coordination

Deep Cuts collaboration uses two separate planes when Codex, Claude, and agy run in separate git
worktrees.

## Problem

Per-agent worktrees solve edit contention for code and docs, but they break coordination if every
MCP server resolves paths from its own checkout:

- `tools/.venv` is gitignored, so linked worktrees may not have a local MCP executable.
- `scratch/coordination` becomes a private mailbox per worktree instead of one shared collab queue.
- `scratch/ccrep.db` becomes a private CCREP ledger per worktree, so proposals, critiques, and
  approvals do not compose.
- Live split logs such as `session.codex.md` are not visible to peers if they are written only into
  each bot's branch and require git merges before every handoff.

## Accepted Scheme

Use isolated worktrees for deliverables and one shared local coordination plane for messages,
quality-ratchet state, and live session logs.

```text
/Users/rlupi/src/deep-cuts          canonical repo and shared coordination plane
/Users/rlupi/src/deep-cuts-codex    Codex worktree, branch bot/codex
/Users/rlupi/src/deep-cuts-claude   Claude worktree, branch bot/claude
/Users/rlupi/src/deep-cuts-agy      agy worktree, branch bot/agy
```

The coordination plane lives under the canonical repo:

```text
/Users/rlupi/src/deep-cuts/scratch/coordination      collab mailboxes and task queue
/Users/rlupi/src/deep-cuts/scratch/ccrep.db          CCREP evidence ledger
/Users/rlupi/src/deep-cuts/doc/collab/sessions/...   live session.<actor>.md logs
```

Each bot still edits deliverables in its own worktree and branch. The shared plane is only for
coordination records.

## MCP Launching

The committed `.mcp.json` launches wrapper scripts:

```json
{
  "mcpServers": {
    "collab": { "command": "tools/run_collab_mcp.py" },
    "ccrep": { "command": "tools/run_ccrep_mcp.py" }
  }
}
```

The wrappers discover the canonical repo with:

```bash
git rev-parse --path-format=absolute --git-common-dir
```

For linked worktrees, the common git directory is the main repo's `.git`, so the wrappers can set
shared defaults without hard-coding Roberto's path in `.mcp.json`:

- `COLLAB_ROOT=<canonical>/scratch/coordination`
- `CCREP_DB=<canonical>/scratch/ccrep.db`
- `CCREP_REPO_ROOT=<canonical>`

Explicit environment variables still override these defaults for unusual setups.

## Live Session Logs

During active collaboration, write live log entries to the canonical repo, not to the bot worktree:

```text
<canonical>/doc/collab/sessions/YYYY-MM-DD-topic/session.codex.md
<canonical>/doc/collab/sessions/YYYY-MM-DD-topic/session.claude.md
<canonical>/doc/collab/sessions/YYYY-MM-DD-topic/session.agy.md
<canonical>/doc/collab/sessions/YYYY-MM-DD-topic/session.roberto.md
```

Each actor writes only its own file. `session.md` is generated by `tools/merge_sessions.py` at
closeout or when `--check` reports stale output. Do not regenerate it during every turn.

## Handoffs

Use collab MCP for handoffs, ACKs, and task claims. Include branch and commit information when the
handoff is about a deliverable, but do not require a commit before ordinary coordination handoffs.
The repository rule still applies: agents do not commit unless Roberto explicitly asks.

CCREP is the quality gate for a concrete artifact:

1. A bot proposes a branch commit.
2. A reviewer evaluates that immutable commit in an isolated worktree.
3. Independent approval plus green checks plus no blocking critiques yields consensus.
4. Merge remains Roberto-gated when the change is sensitive or when the repo's no-commit rule
   applies.

## Rejected Default

Do not use "commit before every handoff, then merge the peer branch before reading the turn" as the
default live-coordination protocol. It couples the coordination log to git history, creates merge
churn on every turn, and conflicts with the project instruction to avoid commits until Roberto asks.

Branch merging remains appropriate for reviewing or integrating deliverables, not for routine
message passing.
